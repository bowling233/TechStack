# 代码风格和编辑器

代码风格很重要，但我认为没有必要花费太多时间配置各种格式化程序、插件等。现在有不少格式化程序试图尽可能减少用户配置（比如 Prettier），我会尽量选择这类格式化程序。

我的配置文件可以在 [:material-github:bowling233/dotfiles](https://github.com/bowling233/dotfiles) 找到。

## EditorConfig

对 EditorConfig 提供无缝支持的有：GitHub、GitLab、IntelliJ IDEA、PyCharm、Vim、Visual Studio 等。**VSCode 需要安装插件。**

对公共项目来说，使用 EditorConfig 是保持代码风格统一的最便捷的方式。

??? example "我的配置文件"

    ```text
    # Located at the project root
    root = true

    # Base settings for most files
    # Linux Kernel Style
    [*]
    charset = utf-8
    end_of_line = lf
    insert_final_newline = true
    indent_style = tab
    indent_size = 8
    trim_trailing_whitespace = true

    # General 4-space files
    [*.{py,json}]
    indent_style = space
    indent_size = 4

    # General 2-space files
    [*.{yaml,clang-format}]
    indent_style = space
    indent_size = 2

    [*.md]
    indent_style = space
    indent_size = 2

    [Makefile]
    indent_style = tab
    ```

## 代码风格

代码风格因编程语言和个人经验的差异极大。

[:material-youtube:CodeAesthetic](https://www.youtube.com/@CodeAesthetic) 频道介绍了一些较高抽象层次的代码组织方法，在大型项目中十分有用。

### C

进行 C 编程时我偏向于遵守内核代码风格，符合偏向底层的思维。有几个要点：

- 缩进
    - 使用 Tab，长度设置等价于 8 个空格。
- 断行
    - 80 字符断行，特殊情况（`grep` 等）除外。
- 大括号换行
    - 函数换。
    - 复合语句不换。
- 命名
    - 本地变量尽量短：`cntusr()`。
    - 全局变量具有描述性：`count_active_users()`。
- 不使用 `typedef`。
- 函数
    - 一个函数的功能保持在一个高中生能理解的范围。
    - 本地变量在 5-10 个左右。
    - 使用 `goto` 集中处理函数退出的过程。
- 注释
    - 代码是显然的，不需要解释的。函数内不写注释。
    - Doxygen 风格，描述代码做了什么。
